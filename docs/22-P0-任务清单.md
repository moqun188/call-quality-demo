### P0-001: 修复 statsManager NaN 问题

**任务 ID**: P0-001  
**优先级**: 🔴 紧急  
**预估工时**: 1h  
**负责人**: 待分配  
**状态**: ❌ 待开始  

**问题描述**:
`statsManager.js` 中的 `getStats()` 方法计算平均维度分数时，由于 `dimensions[k]` 是对象而非数字，导致计算结果为 NaN。

**问题代码**:
```javascript
// 文件: pipeline/statsManager.js, 第 97 行
avgDimensions[k] = Math.round(
  inspections.reduce((s, r) => s + (r.dimensions[k] || 0), 0)  
  // 问题：r.dimensions[k] 是对象 { score, reason }，不是数字！
  / total * 10
) / 10;
```

**修复方案**:
```javascript
avgDimensions[k] = Math.round(
  inspections.reduce((s, r) => s + (r.dimensions[k]?.score || 0), 0)  
  / total * 10
) / 10;
```

**相关文件**:
- [pipeline/statsManager.js](file:///d:/tracecode/pipeline/statsManager.js#L97)
- [pipeline/statsManager.js](file:///d:/tracecode/pipeline/statsManager.js#L94)

3. ✅ 访问 `/api/stats` 接口返回正确的 `avgDimensions`（非 NaN）
4. ✅ 单元测试覆盖 `getStats()` 方法（如有测试框架）

### P0-002: 清理重复 API 调用

**任务 ID**: P0-002  
**优先级**: 🔴 紧急  
**预估工时**: 2h  
**负责人**: 待分配  
**状态**: ❌ 待开始  

**问题描述**:
当前质检流程中，ASR、说话人分离、情绪分析三个模块各自独立调用小米 MiMo API，每次都需要重新编码音频文件，导致：
- Credits 消耗 3 倍
- 质检耗时过长（15-30 秒）
- 重复的文件 IO

**当前流程**:
```
ASR → 调用 MiMo API（音频编码 + HTTP 请求）
说话人分离 → 再次调用 MiMo API（再次编码 + 请求）
情绪分析 → 第三次调用 MiMo API（再次编码 + 请求）
通话总结 → 第四次 API（纯文本）
```

**优化目标**:
合并为单次多模态 API 调用，使用 `mimo-v2.5-pro` 一次性产出所有结果。

**设计方案**:
```javascript
// 新增 pipeline/multimodalAnalyzer.js
async function analyze(audioPath, utterances) {
  const prompt = `请分析这段客服录音，返回以下 JSON 格式：
{
  "transcript": [
    {"speaker": "agent"|"customer", "start": 秒, "end": 秒, "text": "文本", "emotion": "平静|愤怒|..."}
  ],
  "summary": {"callPurpose": "...", "keyPoints": [...], ...}
}`;
  
  // 单次 API 调用
  const result = await callMiMo(audioPath, prompt);
  return parseResult(result);
}
```

- [pipeline/diarization.js](file:///d:/tracecode/pipeline/diarization.js)
- [pipeline/emotion.js](file:///d:/tracecode/pipeline/emotion.js)

**验收标准**:
1. ✅ 质检流程仅调用 1 次 MiMo API（音频）+ 1 次（文本总结）
2. ✅ 质检耗时从 15-30 秒降低到 3-8 秒
3. ✅ 输出格式与原有格式兼容（utterances, emotion, summary）

### P0-003: 修复 uploads 目录清理

**任务 ID**: P0-003  
**优先级**: 🟠 重要  
**预估工时**: 1h  
**负责人**: 待分配  
**状态**: ❌ 待开始  

**问题描述**:
上传的 `.m4a`、`.mp3`、`.converted.mp3` 文件永久堆积在 `uploads/` 目录，占用磁盘空间。

**当前行为**:
- 上传文件保留在 `uploads/`
- 转换后的文件保留在 `uploads/`
- 无自动清理机制

**设计要求**:
- 质检成功后：删除原始上传文件
- 质检失败后：保留文件用于排查（可手动清理）
- 转换后的文件：在质检成功后删除

**实现方案**:
```javascript
// server.js - 质检成功后清理
app.post("/api/inspect", upload.single("audio"), async (req, res) => {
  // ... 质检逻辑 ...
  
  if (result.success) {
    // 清理原始文件
    fs.unlinkSync(req.file.path);
    // 清理转换后的文件
    if (preprocResult.convertedPath && preprocResult.convertedPath !== req.file.path) {
      fs.unlinkSync(preprocResult.convertedPath);
    }
  }
});
```

**相关文件**:
- [server.js](file:///d:/tracecode/server.js#L66)
- [pipeline/preprocessor.js](file:///d:/tracecode/preprocessor.js)

**验收标准**:
1. ✅ 质检成功后，`uploads/` 目录中无残留文件
2. ✅ 质检失败后，文件保留在 `uploads/`（用于排查）
3. ✅ 转换后的 `.converted.mp3` 文件在质检成功后删除
4. ✅ 并发上传时，清理逻辑不影响其他正在处理的文件
