## 📋 敏捷框架

### Sprint 周期

```
周一 ────────────────────────────────────────────── 周五
  │                                               │
  │  Sprint Planning  │  开发  │  Review  │  Retro │
  │    (30min)        │ (3天)  │  (30min) │ (20min)│
  │                   │        │          │        │
```

### 固定活动

| 活动 | 频率 | 时长 | 参与者 | 说明 |
|------|------|------|--------|------|
| **Sprint Planning** | 每周一 | 30min | 全员 | 计划本周任务，分配 Story Points |
| **Daily Standup** | 每天 | 5-10min | 全员 | 昨天做了什么/今天做什么/阻塞什么 |
| **Sprint Review** | 每周五 | 30min | 全员 | 演示本周完成的功能 |
| **Sprint Retro** | 每周五 | 20min | 全员 | 回顾改进点 |
| **Backlog Grooming** | 每周三 | 15min | PM | 整理和排序 Backlog |

---

## 📝 用户故事 (User Story) 格式

```
US-XXX: <简短标题>
- 作为 <角色>, 我想要 <功能>, 以便 <价值>
- 验收标准:
  1. AC1: ...
  2. AC2: ...
- 故事点: 1/2/3/5/8/13
- 优先级: P0/P1/P2/P3
- 负责人: xxx
- Sprint: Sx
```

### 故事点参考

| 故事点 | 含义 | 示例 |
|--------|------|------|
| 1 | 极简改动 | 修改一个配置值 |
| 2 | 简单任务 | 修复一个明确的 Bug |
| 3 | 中等任务 | 新增一个 API 接口 |
| 5 | 较复杂任务 | 重构一个模块 |
| 8 | 复杂任务 | 引入新框架/新模块 |
| 13 | 非常复杂 | 前后端分离/数据库迁移 |

## 📌 定义完成 (Definition of Done)

一个 User Story 完成的标准：

1. ✅ 代码编写完成
2. ✅ 单元测试通过（如有测试框架）
3. ✅ 代码 Review 通过（至少 1 人）
4. ✅ 功能手动验证通过
5. ✅ 文档已更新（如需要）
6. ✅ 已合并到 dev 分支
