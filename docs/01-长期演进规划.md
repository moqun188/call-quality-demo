# 🚀 长期演进规划（长期路线图）

> 更新日期: 2026-06-17
> 维护者: moqun188 / 协作成员（2人协作）
> 目标: 从 MVP 走向生产级可落地的质检系统

---

## 1. 规划目标与原则

- 以“真实录音质检的可落地体验”为核心目标：从单文件质检，扩展到批量/流式场景，逐步引入可观测性与合规能力。
- 稳定性优先于新特性：P0 修复优先，再推进架构升级。
- 以文档为骨架驱动迭代：所有变动都先写清楚再改代码，确保协作时快速对齐。

---

## 2. 里程碑划分（P0-P3）

### P0：紧急修复与最小可用落地（1-2 周内）

目标：解决现有阻断问题，建立稳定的开发循环。

关键任务：
- 修复 statsManager 的 NaN bug（avgDimensions）并增强单元测试
- 复用已转码音频，减少重复 API 调用（ASR / 情绪 / 说话人合并为单次调用）
- 修复 uploads 清理策略：质检成功后清理原始文件，失败时保留可用片段以排查
- 日志轮转：按大小滚动，7 天清理，记录 sessionId
- 完成初步的 P0 文档化，确保新成员能快速上手

> **详细任务分解请查看 [docs/22-P0-任务清单.md](./22-P0-任务清单.md)**

交付物：
- bugfix/stats-nan 分支（4 行级改动）
- statsManager 修改：avgDimensions 使用 score 字段
- 更新 docs/01-长期演进规划.md（本文件

）的 P0 清单

---

### P1：稳定性与工程化基础（第 1 个月内）

目标：建立可持续的协作节奏、可测试性和可观测性。

关键任务：
- 质检规则可配置化：外部规则集，便于新场景扩展
- 数据持久化：从 JSON 文件切换到 SQLite，便于查询与分析
- 测试框架：Jest 覆盖关键模块，单元测试覆盖路径>60%
- 代码质量工具：ESLint + Prettier，统一风格
- CI/CD 基础：GitHub Actions 自动测试、基本的代码检查、提交规范
- 监控与日志：结构化日志、基本指标、错误聚合
- 批量质检队列：支持小规模并发

交付物：
- README 更新，指向 docs/01-长期演进规划.md
- 新增 1-2 个任务清单，分解到可执行的工作项
- docs/02-架构设计/11-分支策略 等文档完善

---

### P2：架构演进与能力提升（第 2-3 个月）

目标：从单体 MVP 向可扩展、可维护的生产级架构演进。

关键任务：
- 前后端分离，考虑 Vue 3 + Vite，提升前端开发体验
- TypeScript 引入，核心模块强类型
- 知识库增强：向量数据库+规则匹配，提升质检准确性
- 多用户与权限：JWT/登录页、管理员/质检员/查看者角色
- 安全合规：敏感信息脱敏、数据访问审计

交付物：
- 架构文档更新（docs/02-架构设计.md
）
- 设计阶段的里程碑评估

---

### P3：生产就绪与规模化（第 4-6 个月及以后）

目标：达到可对外提供的生产级质检能力。

关键任务：
- 接入真实客服录音系统（CRM/工单系统对接）
- 实时/流式质检能力（流式 ASR、实时评分）
- 质检人员工作台：审核、修订、批注
- AI 辅助修正：人工修订数据回流用于微调训练
- 数据看板：趋势、高频问题、改进建议
- Docker/容器化部署，CI/CD 全覆盖

交付物：
- 生产就绪文档（部署、运维、数据迁移）
- 多人协作下的 PR 规范与评审流程

---

## 3. 具体实施要点

- 文档先行：任何架构/流程改动，先在 docs/ 写清楚再实现代码。
- 里程碑评估：每个阶段结束进行回顾，确认进度与风险。
- 任务分解：把大任务拆成小任务，明确负责人与验收标准。
- 质量基线：每个 PR 关联一个验收清单，确保新功能不破坏现有流程。
