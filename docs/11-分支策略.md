# 🌿 分支策略

> Git Flow 简化版，适配 2~3 人小团队协作
> 核心理念：main 永远稳定，dev 用于集成，feature 用于开发

---

## 📐 分支模型

```
main  (生产分支，受保护)
  ↑
  │      Pull Request（代码评审后合并
  │                 ↗
  │         ↗   ↗
 dev   (集成开发分支)
  │
  ├─ feature/asr-optimize     (ASR 优化)
  ├─ feature/storage-upgrade  (数据库接入)
  ├─ bugfix/stats-nan         (Bug 修复)
  └─ refactor/pipeline-clean  (重构)
```

### 分支类型

| 类型 | 命名前缀 | 从哪个分支创建 | 合并到哪个分支 | 生命周期 |
|------|---------|---------------|---------------|---------|
| **main** | - | - | - | 永久 |
| **dev** | - | main | main | 永久 |
| feature | `feature/` | dev | dev | 短（数小时~1周） |
| bugfix | `bugfix/` | dev | dev | 短（数小时~1天） |
| hotfix | `hotfix/` | main | main + dev | 极短（紧急） |
| refactor | `refactor/` | dev | dev | 中（1~2周） |

## 🔄 开发流程

### 日常开发步骤（标准流程）

```
1. 切换到 dev 并拉取最新
   git checkout dev
   git pull origin dev

2. 创建功能分支
   git checkout -b feature/你的功能名

3. 开发并提交（频繁小步提交）
   git add .
   git commit -m "feat: 你的功能描述"

4. 推送分支到远程
   git push origin feature/你的功能名

5. 在 GitHub 创建 Pull Request
   base: dev
   compare: feature/你的功能名

6. 代码评审（至少 1 人通过）

7. 合并到 dev（建议 Squash and merge）

8. 删除已合并的分支
   git branch -d feature/你的功能名
   git push origin --delete feature/你的功能名
```

### Bug 修复流程

```
1. 创建修复分支
   git checkout -b bugfix/stats-nan

2. 修复 + 提交
   git commit -m "fix: 修复 statsManager NaN 问题"

3. PR 到 dev

4. 紧急 Bug：同时 cherry-pick 到 main
   git checkout main
   git cherry-pick <commit-hash>
   git push origin main
```

## 📋 分支命名规范

### 命名模板
```
<类型>/<简短描述>

类型: feature | bugfix | hotfix | refactor | chore | docs

简短描述:
- 使用英文（或拼音，不强制）
- 用 kebab-case 格式（短横线分隔）
- 不超过 30 字符
- 见名知意

```

### ✅ 推荐命名
```
feature/asr-cache              (ASR 缓存功能)
bugfix/stats-nan               (修复 NaN Bug)
refactor/pipeline-clean        (重构 pipeline 清理)
chore/update-dep               (更新依赖)
docs/add-roadmap               (更新文档)
```

### ❌ 不推荐命名
```
test           (含义不清
fix           (太笼统
tmp            (临时性命名，容易遗忘
dev2          (分支管理混乱)
```

## 🚀 版本发布流程

### P1 阶段完成后，发布 v0.3.0

```
1. 在 dev 完成所有功能，测试通过

2. 创建 release 分支
   git checkout -b release/v0.3.0

3. 版本号更新
   - package.json
   - 生成 changelog

4. 合并到 main
   git checkout main
   git merge release/v0.3.0

5. 打 tag
   git tag -a v0.3.0 -m "Release v0.3.0"
   git push origin v0.3.0

6. 合并回 dev（确保 dev 也包含 release 的变更）
   git checkout dev
   git merge main
```

## 🔒 分支保护规则

### main 分支
- ❌ 禁止直接 push
- ✅ 必须通过 PR 合并
- ✅ 需要至少 1 人 review 通过
- ✅ 必须通过 CI 测试

### dev 分支
- ⚠️ 不鼓励直接 push
- ✅ 推荐通过 PR 合并
- ⚠️ 紧急情况下可直接 push（需写注释）

## 🤝 协作约定

### Pull Request 规范
1. **标题**：使用 [13-提交规范.md](./13-提交规范.md) 的格式
2. **描述**：说明做了什么、为什么、如何测试
3. **关联 Issue**：`Fixes #123`（自动关闭 Issue）
4. **Assignees**：至少 1 个 reviewer
5. **Labels**：添加合适的标签（bug / feature / refactor / docs）

### 代码评审清单
详见 [14-代码评审指南.md](./14-代码评审指南.md)

### 冲突处理
- 遇到 merge conflict 时，不要强行合并
- 与相关开发人员协商解决
- 如需帮助，在 PR 评论中 @ 其他人

## 💡 快速参考

### 我要修复 Bug
```bash
git checkout dev
git pull origin dev
git checkout -b bugfix/xxx
# 修改代码
git commit -m "fix: 修复 xxx"
git push origin bugfix/xxx
# GitHub 创建 PR
```

### 我要开发新功能
```bash
git checkout dev
git pull origin dev
git checkout -b feature/xxx
# 开发
git commit -m "feat: 实现 xxx"
git push origin feature/xxx
# GitHub 创建 PR
```

### 我要同步最新代码
```bash
git checkout dev
git pull origin dev
git checkout feature/xxx
git rebase dev    # 或 git merge dev
```
