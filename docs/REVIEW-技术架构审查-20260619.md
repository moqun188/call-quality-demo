## 一、架构评价

### 整体架构：✅ 合理

```
server.js (Express)
  └── pipeline/index.js (编排器)
        ├── preprocessor.js (音频预处理)
        ├── multimodalAnalyzer.js (多模态单次调用)
        ├── asr.js (ASR 转写)
        ├── diarization.js (说话人分离)
        ├── emotion.js (情绪分析)
        ├── qualityAnalyzer.js (质检评分)
        └── summarizer.js (通话总结)
  └── pipeline/statsManager.js → database.js (SQLite)
```

**优点**：
- Pipeline 模式清晰，各模块职责单一
- 多模态优先、独立模块回退的降级策略设计优秀
- SQLite + WAL 模式选择正确

**问题**：见下方详细分析

## 二、代码问题清单（按严重度排序）

### 🔴 P0 — 必须修复

#### 1. `server.js` — API Key 泄露风险

```javascript
// server.js 第 ~185 行
requestHeaders: {
  "api-key": apiKey ? `${apiKey.substring(0, 8)}...` : "未配置",
},
```

**问题**：`/api/test/asr` 接口在响应中返回了 API Key 前 8 位。虽然做了截断，但在日志或错误响应中可能泄露更多信息。

**建议**：完全移除 API Key 相关字段的输出，只返回 `apiKeyConfigured: true/false`。

---

#### 2. `server.js` — 错误堆栈泄露

```javascript
// server.js /api/test/asr 的 catch 块
res.status(500).json({
  success: false,
  error: err.message,
  stack: err.stack,  // ← 生产环境禁止返回堆栈
});
```

**问题**：`err.stack` 包含文件路径、代码行号等内部信息，直接返回给客户端是安全隐患。

**修复**：生产环境只返回 `err.message`，`stack` 仅写日志。

---

#### 3. `multimodalAnalyzer.js` — API 返回格式未验证

```javascript
// _parseResult 中
const utterances = data.transcript.map((seg, idx) => {
  const role = seg.role === "agent" || seg.role === "customer" ? seg.role : "customer";
```

**问题**：如果 `data.transcript` 中某个元素缺少 `text` 字段，后续 `qualityAnalyzer.js` 的 `u.text.join("")` 会抛异常。没有对 `seg.text` 做非空校验。

**修复**：`text: seg.text || ""` 已有，但应在 map 前过滤掉 `text` 为空的无效条目。

---

#### 4. `statsManager.js` — 并发写入竞态

```javascript
function addInspection(result) {
  const record = { id: nextInspectionId(), ... };
  stmts.insertInspection.run(...);
}
```

**问题**：`nextInspectionId()` 和 `insertInspection.run()` 之间没有事务保护。如果两个请求同时调用，可能产生 ID 冲突。

**修复**：better-sqlite3 支持事务，将 ID 生成和插入包裹在 `db.transaction()` 中。或者直接用 SQLite 的 `AUTOINCREMENT`，移除手动 ID 计算。

---

### 🟠 P1 — 应该修复

#### 5. `qualityAnalyzer.js` — 评分逻辑过于简单

```javascript
_evalCompliance(agentTexts, fullText) {
  let score = 7;  // 基准分固定为 7
  if (hasOpening) score += 1;
  if (hasClosing) score += 1;
  if (politeCount >= 4) score += 1;
  return { score: Math.min(10, score), ... };
}
```

**问题**：
- 基准分固定 7 分，意味着即使客服完全不说话，也能拿 70 分
- 关键词匹配过于粗糙（`fullText.includes("您好")` 就算合规）
- 没有权重区分：说 1 次"您好"和全程礼貌用语，得分一样

**建议**：
- 基准分改为 5，拉开差距
- 引入关键词出现频次加权
- 增加负面行为扣分（如长时间沉默、打断客户）

---

#### 6. `diarization.js` — 角色检测关键词需扩充

```javascript
const agentPattern = /请问|欢迎致电|您好.*客服|帮您|为您|.../;
const custPattern = /我想|我要|咨询|投诉|多少钱|.../;
```

**问题**：
- 关键词列表覆盖不全（金融、医疗、电商等行业术语缺失）
- 正则匹配无权重，"您好"和"工号"权重应该不同
- 当两个说话人得分相同时，兜底策略（"您"的频率）不够可靠

**建议**：
- 将关键词配置化，按行业加载不同词库
- 引入 TF-IDF 或简单权重系统
- 增加置信度输出，低置信度时标记 `role: "unknown"`

---

#### 7. `summarizer.js` — Mock 模式有重复逻辑

```javascript
_mockSummary() {
  // ...
  let callPurpose = "常规咨询";
  if (/退款|退货/.test(fullText)) callPurpose = "退款/退货处理";
  // ...
}
```

**问题**：`_mockSummary` 和 `_realSummary` 的返回结构不完全一致。`_mockSummary` 缺少 `raw` 字段，调用方如果访问 `result.raw` 会得到 `undefined`。

**修复**：统一返回结构，mock 模式也填充 `raw: "(mock)"`。

---

#### 8. `logger.js` — `appendFileSync` 性能问题

```javascript
if (this.logToFile) {
  fs.appendFileSync(logFilePath, logMessage + "\n");
}
```

**问题**：每次日志写入都是同步文件操作，在高并发场景下会阻塞事件循环。

**建议**：改为异步写入（`fs.appendFile` 或使用 `createWriteStream` + 缓冲区）。

---

### 🟡 P2 — 建议优化

#### 9. `pipeline/index.js` — `_runScenario` 未使用 `notify` 的 async 特性

```javascript
if (onStep) onStep(i, { ...scenarioSteps[i], status: "completed" });
```

**问题**：主流程的 `notify` 是 `async` 并 `await`，但 `_runScenario` 中直接同步调用 `onStep`，不一致。

---

#### 10. `audioConverter.js` — 缺少超时控制

```javascript
const cmd = ffmpeg(inputPath)
  .audioCodec("libmp3lame")
  // ...
  .on("error", (err) => reject(err))
  .on("end", () => resolve(...));
cmd.save(outputPath);
```

**问题**：如果 ffmpeg 卡住（大文件、损坏音频），没有超时机制，会永远 hang。

**建议**：添加 `setTimeout` 强制终止 ffmpeg 进程（如 60 秒超时）。

---

#### 11. `server.js` — 两个 `/api/inspect` 接口功能重复

```javascript
app.post("/api/inspect", upload.single("audio"), async (req, res) => { ... });
app.post("/api/inspect/stream", upload.single("audio"), async (req, res) => { ... });
```

**问题**：两个接口逻辑几乎完全一样，区别只是 `flushHeaders` 和 `flush()`。应合并为一个，通过 query 参数 `?stream=true` 控制。

---

#### 12. `database.js` — 缺少数据清理策略

**问题**：`inspections` 表没有数据清理机制。随着质检次数增长，数据库会持续膨胀。

**建议**：添加定期清理策略（如保留最近 90 天数据），或提供管理员手动清理接口。

## 三、架构优化建议

### 3.1 模块解耦

当前所有模块都直接读取 `process.env`，建议引入配置中心：

```javascript
// config.js
module.exports = {
  mimo: {
    apiKey: process.env.MIMO_API_KEY,
    baseUrl: process.env.MIMO_BASE_URL || "https://api.xiaomimimo.com/v1",
    models: {
      asr: process.env.ASR_MODEL || "mimo-v2.5-asr",
      pro: "mimo-v2.5-pro",
      base: "mimo-v2.5",
    },
  },
  server: { port: process.env.PORT || 3000 },
  quality: { weights: { compliance: 0.30, knowledge: 0.25, process: 0.25, communication: 0.20 } },
};
```

### 3.2 错误处理统一

当前各模块的错误处理风格不一致：
- `asr.js`：catch → 回退 mock
- `diarization.js`：catch → 回退 mock
- `multimodalAnalyzer.js`：catch → 返回 null（由 pipeline 回退）
- `summarizer.js`：catch → 回退 mock

**建议**：定义统一的 `PipelineError` 类，各模块 throw，由 pipeline 编排器统一处理回退逻辑。

### 3.3 测试覆盖

当前 **零测试**。P0 优先补充：
1. `qualityAnalyzer.js` — 纯逻辑，最容易写单元测试
2. `statsManager.js` — 验证 NaN 修复
3. `diarization.js` 的 `_detectRoles` — 关键词匹配逻辑
